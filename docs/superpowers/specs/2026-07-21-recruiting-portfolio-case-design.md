# AI 知识博主内容运营投递作品设计

## 目标

为 AI 知识博主内容运营岗位制作一套可以直接放入简历的完整案例。作品以秋芝2046的公开内容为研究样本，但定位为个人项目，不代表官方合作、真实运营或真实后台数据。

最终交付包含独立网页 `/portfolio` 和一份 6—8 页 PDF。招聘方应能在 3 分钟内理解项目价值，也能继续查看完整内容执行细节。

## 页面结构

### 1. 三分钟项目摘要

- 项目名称：AI 知识博主内容运营作品集——以秋芝2046公开内容为研究样本。
- 说明项目背景、个人角色、研究范围和六项交付成果。
- 提供“下载 PDF”和“查看公开研究页”两个入口。
- 明确标注个人项目、非官方、反馈与指标均为模拟数据。

### 2. 账号观察

- 用一屏说明目标受众、内容特点和内容机会点。
- 结论来源于现有 30 条代表视频分析，不重复展示全部视频卡片。
- 区分公开事实、内容分析和待核验信息。

### 3. 三个候选选题

1. 主案例：每天记 3 行，让桌面 Agent 整理日报和周报。
2. 备选：用桌面 Agent 整理会议资料，生成待办清单。
3. 备选：把散落的 AI 学习资料整理成可检索的个人资料库。

每个选题展示目标用户、用户痛点、结果价值、可演示性、账号匹配度和主要风险。主案例给出进入制作的理由，另两个作为后续选题储备。

### 4. 主案例完整内容包

- B站：三个标题、封面文案、章节结构和完整口播稿。
- 小红书：标题、八页图文逐页内容和配套正文。
- 抖音：独立 60 秒脚本与镜头建议。
- 视频号：独立 60 秒脚本与转发场景设计。
- 公众号：标题、摘要和完整文章提纲。
- 所有平台共享同一事实底座，但开场、结构和行动引导分别适配。

较长内容使用标签页和展开区域，默认只展示摘要，避免招聘方首次打开时面对过长页面。

### 5. 七天发布排期

使用统一内容编号关联 D-2 预热、D-1 预告、D0 首发、D1/D2 二次分发、D3 评论维护和 D7 复盘。每个节点包含平台、交付物、负责人角色、依赖、状态和风险。

### 6. 模拟用户反馈

- 展示 12—15 条匿名模拟评论。
- 包含去重组、平台、匿名文本、情绪、需求标签、P0—P3 优先级、是否需要回复、状态和下一步动作。
- 明确说明这些记录只用于展示反馈整理方法，不是实际评论或私信。
- 隐私、安全和事实风险优先处理；不推断用户身份或敏感属性。

### 7. 复盘思路

- 分开目标、观察指标、成功标准、异常信号和解释假设。
- 指标包括点击、完播、收藏、评论结构、资料需求和高优先级反馈。
- 不使用虚构结果证明内容成功；采用“发布后将如何判断”的前瞻复盘框架。
- 给出保留、停止、尝试和单变量实验建议。

## 内容与数据组织

网页和 PDF 使用同一套经过审核的结构化内容，避免两份作品出现不同数字、标题或结论。现有案例草案可以复用，但必须补齐完整 B站口播稿、逐页小红书文案、反馈字段和前瞻式成功标准。

所有动态产品能力、平台规则和公开账号数据在使用前重新核验并记录查询日期；无法核验的信息标为待核验，不写成事实。

## 交互与视觉

- `/portfolio` 使用现有黄色、黑色、米白品牌体系，但比首页更像招聘作品文档。
- 页面采用单列阅读主线，关键数据使用卡片、时间线和表格。
- 平台内容通过标签切换，完整稿使用可展开区域。
- 手机端保持单列，无横向溢出；反馈表在窄屏转为卡片。
- PDF 控制在 6—8 页，保留项目摘要、关键内容样稿、排期、反馈和复盘，不塞入全部长稿。

## 路由与边界

- 新增 `/portfolio`，不改变现有公开首页的“账号洞察、视频速览”结构。
- 现有 `/ops` 工作台继续保留，但不在投递作品页公开入口。
- 不自动发布、回复、评论、点赞或发送私信。
- 不声称与秋芝2046存在合作关系，不使用真实后台数据或真实用户信息。

## 验收标准

- 招聘方可在 3 分钟内看懂项目目标、方法和六项交付成果。
- 三个选题与主案例完整内容包均可从页面查看。
- 七天排期字段完整，12—15条模拟反馈包含去重、标签、优先级、回复判断和下一步。
- 复盘采用成功标准与下一轮实验，不把模拟指标写成真实结果。
- PDF 可下载，内容与网页一致，控制在 6—8 页。
- 页面在桌面端和手机端正常显示，标签页、展开区域、PDF链接和外部链接可用。
- 现有首页、视频筛选、`/ops` 鉴权和数据接口测试继续通过。
- 完成后同步发布到现有网站和 GitHub 公开仓库。
