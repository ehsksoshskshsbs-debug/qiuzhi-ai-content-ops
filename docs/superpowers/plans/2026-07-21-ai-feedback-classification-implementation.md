# AI 反馈分类实施计划

## 交付目标

在现有云端反馈工作台中加入“AI 建议、人工确认”的分类流程。AI 只读取经过隐私遮蔽的反馈摘要，只建议需求标签与优先级；未经人工接受或修改，不更新正式字段。

## 任务 1：数据模型与迁移

- 为反馈记录增加 AI 建议、理由、置信度、模型、分类时间与审核状态字段。
- 编写 D1/Drizzle 迁移，并兼容已有数据库实例。
- 扩展领域类型、查询和更新映射，确保列表与详情接口返回 AI 审核状态。

验收：旧记录可正常读取；新字段默认不影响现有人工录入和筛选。

## 任务 2：服务端分类器

- 使用 OpenAI Responses API 与严格 JSON Schema 输出。
- 默认使用适合高频分类的 `gpt-5.6-luna`，允许用环境变量覆盖。
- 仅向模型发送服务端读取且已遮蔽敏感信息的摘要。
- 校验拒答、超时、不完整响应、非法枚举和低置信度等边界情况。
- 不在日志或响应中暴露 API 密钥与原始敏感数据。

验收：给定摘要可稳定返回需求标签、优先级、简短理由和 0–1 置信度；失败时可继续人工处理。

## 任务 3：分类与人工审核 API

- 增加单条反馈 AI 分类接口。
- 增加人工接受或修改建议的审核接口。
- 未审核时只保存建议字段；审核后才更新正式需求标签和优先级。
- 记录“接受 AI 分类”或“修改 AI 分类”的审计事件与审核人。
- 增加基础频率限制和并发分类保护。

验收：AI 不能直接修改正式字段；每次人工确认都有可追踪记录。

## 任务 4：工作台交互

- 在反馈详情编辑器中增加 AI 分类卡片。
- 提供“获取 AI 建议”、建议结果、理由、置信度和审核状态。
- 允许人工调整标签和优先级后保存。
- 对 P0 和低置信度建议显示明确提醒。
- 新建记录需先保存，再进行 AI 分类。

验收：用户可在一个页面完成“生成建议 → 人工复核 → 保存正式字段”。

## 任务 5：验证与发布

- 使用模拟 Responses API 覆盖成功、拒答、超时和非法响应测试。
- 覆盖“未经审核不更新正式字段”“接受/修改后写审计事件”等集成场景。
- 运行 lint、类型检查、测试、数据库迁移生成和生产构建。
- 安全配置生产环境密钥与模型变量，保存并发布新的 Sites 版本。
- 发布后检查登录保护、工作台加载、分类失败回退与数据持久化。

验收：自动化检查通过，生产工作台可完成人工确认式 AI 分类。

## 实施限制

- 不做无人值守批量分类。
- 不自动回复、分配负责人、改变处理状态或向外部平台写入数据。
- 不把评论或私信中的可识别个人信息发送给模型。
- 不在客户端代码、Git、日志或错误消息中保存或显示密钥。
- 模型建议是辅助判断，最终责任由人工确认环节承担。
